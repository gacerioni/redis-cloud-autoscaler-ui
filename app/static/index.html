The autoscaler is scale-up only

Prometheus scrapes the database's native metrics endpoint (:8070) and evaluates alert rules. When usage crosses the threshold, Alertmanager fires a webhook to the autoscaler, which looks up the matching scaling rule for that database and calls the Redis Cloud REST API to apply the new size.

Scale-down is intentionally NOT reactive — yo-yo behavior would destabilize production. Real customers schedule the return to baseline after the event.

What's new in Redis Cloud Pro

Atomic Slot Migration (ASM) and auto-shard rebalancing replace the old "double the shards" approach with granular slot-level moves. Result: smaller scale-up steps, real scale-down, and lower latency impact during the migration.

How this works in production: reactive scale-down is dangerous — oscillating traffic causes yo-yo scaling and destabilizes the cluster. Real customers use either a manual trigger after the event or a long scheduled window (typically ≥ 3 hours) via cron. For the demo we use a short 5 minute window so the run is reproducible — the autoscaler itself is scale-up only.

⚙ Admin tools danger zone

use between demo runs to reset state
Flush database
Wipes all keys (FLUSHDB). Use after a "Memory fill" preset to bring memory usage back to ~0.
Reset DB to baseline
Force-scales the DB back to 1,000 ops/sec · 2 GB via the Redis Cloud REST API (bypasses the autoscaler — instant reset for the next demo).
Reload scaling rules
Re-registers the 4 autoscaler rules (Throughput up/down, Memory up/down). Use if the autoscaler returns 500 on alerts or you accidentally wiped its rule storage.
